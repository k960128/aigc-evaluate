# L2 知识库整改流程图

本文档描述本次整改的闭环：把 L2 从“可运行 Mock 链路”推进到“可维护、可同步、可观测的知识库链路”。核心范围包括知识库管理接口、同步事件、索引同步状态回写，以及 MySQL Mock 召回开关。

## 1. 本次整改总览

```mermaid
flowchart TD
    USER["运营/测试/管理员"] --> API["L2KnowledgeBaseController<br/>知识库管理接口"]
    API --> SERVICE["L2KnowledgeBaseService<br/>规则/特征/同步事件编排"]

    SERVICE --> RULE["risk_detail_rule<br/>风险小类判定规则"]
    SERVICE --> FEATURE["risk_attack_feature<br/>攻击/安全例外特征"]
    SERVICE --> EVENT["kb_sync_event<br/>知识库同步事件"]

    EVENT --> SYNC["同步状态更新接口<br/>本阶段用于本地状态流转"]
    SYNC --> EVENT_STATUS["更新 kb_sync_event<br/>ES/Milvus 状态、错误信息、重试次数"]
    EVENT_STATUS --> FEATURE_STATUS["回写 risk_attack_feature<br/>sync_status/es_sync_status/milvus_sync_status"]

    FEATURE --> RECALL_SWITCH{"app.l2.mock-recall-enabled"}
    RECALL_SWITCH -->|true| MYSQL_RECALL["MySqlMockL2RecallClient<br/>从 MySQL 特征轻量召回"]
    RECALL_SWITCH -->|false| EMPTY_RECALL["DefaultL2RecallClient<br/>空召回降级"]

    MYSQL_RECALL --> L2["L2EvaluationService<br/>召回/融合/路由判定"]
    EMPTY_RECALL --> L2
    L2 --> PIPELINE["eval_pipeline_node_detail<br/>记录 L2 节点快照"]
    L2 --> RESULT["样本结果<br/>自动评分/人工核验/失败"]
```

## 2. 规则维护流程

```mermaid
flowchart TD
    START["开始维护风险小类规则"] --> ACTION{"接口动作"}

    ACTION -->|分页/详情| QUERY["按风险大类、小类、状态筛选<br/>返回 L2RiskDetailRuleVO"]

    ACTION -->|新增| CREATE_REQ["SaveL2RiskDetailRuleRequest"]
    ACTION -->|编辑| UPDATE_REQ["SaveL2RiskDetailRuleRequest + id"]
    ACTION -->|启停| STATUS_REQ["UpdateL2KnowledgeStatusRequest"]
    ACTION -->|删除| DELETE_REQ["id"]

    CREATE_REQ --> VALIDATE["校验 risk_details 存在且未删除"]
    UPDATE_REQ --> VALIDATE
    STATUS_REQ --> LOAD_RULE["读取 risk_detail_rule"]
    DELETE_REQ --> LOAD_RULE

    VALIDATE --> UNIQUE["校验同一 riskDetailsId<br/>只保留一条有效规则"]
    UNIQUE --> SAVE_RULE["写入/更新 risk_detail_rule"]
    LOAD_RULE --> UPDATE_RULE["更新状态或逻辑删除 risk_detail_rule"]

    SAVE_RULE --> RULE_EVENT["写入 kb_sync_event<br/>aggregateType=DETAIL_RULE"]
    UPDATE_RULE --> RULE_EVENT
    RULE_EVENT --> END["规则变更闭环完成"]
    QUERY --> END
```

## 3. 攻击特征维护流程

```mermaid
flowchart TD
    START["开始维护攻击特征"] --> ACTION{"接口动作"}

    ACTION -->|分页/详情| QUERY["按风险大类、小类、特征类型、极性、同步状态筛选<br/>返回 L2RiskAttackFeatureVO"]

    ACTION -->|新增| CREATE_REQ["SaveL2RiskAttackFeatureRequest"]
    ACTION -->|编辑| UPDATE_REQ["SaveL2RiskAttackFeatureRequest + id"]
    ACTION -->|启停| STATUS_REQ["UpdateL2KnowledgeStatusRequest"]
    ACTION -->|删除| DELETE_REQ["id"]

    CREATE_REQ --> VALIDATE["校验 risk_details 存在且未删除"]
    UPDATE_REQ --> VALIDATE
    STATUS_REQ --> LOAD_FEATURE["读取 risk_attack_feature"]
    DELETE_REQ --> LOAD_FEATURE

    VALIDATE --> HASH["规范化 featureText<br/>计算 contentHash"]
    HASH --> UNIQUE["校验 contentHash 唯一"]
    UNIQUE --> SAVE_FEATURE["写入/更新 risk_attack_feature<br/>同步状态置为待同步"]

    LOAD_FEATURE --> STATUS_ACTION{"变更类型"}
    STATUS_ACTION -->|启停| MARK_PENDING["更新状态、版本号<br/>同步状态置为待同步"]
    STATUS_ACTION -->|删除| MARK_DELETE["同步状态置为删除待同步<br/>再逻辑删除"]

    SAVE_FEATURE --> FEATURE_EVENT["写入 kb_sync_event<br/>aggregateType=ATTACK_FEATURE"]
    MARK_PENDING --> FEATURE_EVENT
    MARK_DELETE --> FEATURE_EVENT

    FEATURE_EVENT --> END["特征变更闭环完成"]
    QUERY --> END
```

## 4. 同步事件与状态回写流程

```mermaid
flowchart TD
    START["查询待处理同步事件"] --> PAGE["/sync-event/page<br/>按聚合类型、操作类型、状态筛选"]
    PAGE --> EVENT["选择 kb_sync_event"]

    EVENT --> UPDATE["/sync-event/status<br/>写入 ES/Milvus 状态"]
    UPDATE --> EVENT_WRITE["更新事件状态<br/>esStatus、milvusStatus、lastError、retryCount"]

    EVENT_WRITE --> TYPE{"aggregateType"}
    TYPE -->|DETAIL_RULE| ONLY_EVENT["仅保留事件状态<br/>规则不回写索引状态"]
    TYPE -->|ATTACK_FEATURE| BACK_FEATURE["回写 risk_attack_feature<br/>es_sync_status、milvus_sync_status"]

    BACK_FEATURE --> AGG{"聚合同步状态"}
    AGG -->|ES 和 Milvus 均已同步| SYNCED["sync_status=SYNCED"]
    AGG -->|任一失败| FAILED["sync_status=FAILED"]
    AGG -->|删除操作| DELETE_PENDING["sync_status=DELETE_PENDING"]
    AGG -->|其他情况| PENDING["sync_status=PENDING"]

    ONLY_EVENT --> END["同步事件处理完成"]
    SYNCED --> END
    FAILED --> END
    DELETE_PENDING --> END
    PENDING --> END
```

## 5. Mock 召回开关流程

```mermaid
flowchart TD
    START["应用启动"] --> CONFIG["读取 app.l2.mock-recall-enabled"]

    CONFIG --> DECIDE{"是否开启 MySQL Mock 召回"}
    DECIDE -->|true 或未配置| MYSQL_BEAN["注册 MySqlMockL2RecallClient"]
    DECIDE -->|false| DEFAULT_BEAN["注册 DefaultL2RecallClient"]

    MYSQL_BEAN --> MYSQL_FLOW["L2 评测时读取 risk_attack_feature<br/>按文本包含、关键词重合、极性、风险等级生成 Mock 命中"]
    DEFAULT_BEAN --> DEFAULT_FLOW["L2 评测时返回空召回结果<br/>保持降级可启动"]

    MYSQL_FLOW --> L2["L2EvaluationService<br/>统一消费 L2RecallClient"]
    DEFAULT_FLOW --> L2

    L2 --> ROUTE{"L2 路由结果"}
    ROUTE -->|高置信违规| BLOCK["自动判不安全<br/>L2 节点 BLOCKED"]
    ROUTE -->|低风险安全| PASS["自动判安全<br/>L2 节点 PASSED"]
    ROUTE -->|模糊区| REVIEW["进入人工核验<br/>L2 节点 NEED_MANUAL_REVIEW"]
```

## 6. 攻击特征同步状态流转

```mermaid
stateDiagram-v2
    [*] --> PENDING: 新增/编辑/启停
    PENDING --> SYNCED: ES 与 Milvus 均同步成功
    PENDING --> FAILED: ES 或 Milvus 任一同步失败
    FAILED --> PENDING: 重新触发同步
    SYNCED --> PENDING: 再次编辑/启停
    PENDING --> DELETE_PENDING: 删除
    SYNCED --> DELETE_PENDING: 删除
    FAILED --> DELETE_PENDING: 删除
    DELETE_PENDING --> [*]: 逻辑删除后等待后续真实索引删除闭环
```

## 7. 本次整改边界

- MySQL 仍是唯一事实源，ES 和 Milvus 仍未接真实服务。
- 本阶段通过同步事件和状态字段先打通“可观测、可回写”的本地闭环。
- MySQL Mock 召回用于开发验证，可通过 `app.l2.mock-recall-enabled` 切换。
- L2 主流程继续通过 `L2RecallClient` 抽象消费召回结果，后续替换真实 ES/Milvus 客户端时不需要改 L2 主流程。
- L3 Judge 暂未进入本次整改范围，模糊区仍进入人工核验。
