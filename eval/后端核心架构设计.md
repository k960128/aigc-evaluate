# 大模型安全评测平台 - 后端核心架构设计 (V1.0)

## 1. 架构总览与技术栈

本系统定位于**企业级、高并发的大模型安全评测网关**。核心目标是在应对外部大模型 API 严苛的限流（QPS/TPM）条件下，实现海量评测任务的高效调度，并通过多级漏斗大幅降低裁判大模型（Judge LLM）的算力成本。

### 1.1 核心技术栈

* **底层计算:** Java 21 (强制开启虚拟线程 Virtual Threads)
* **框架:** Spring Boot 3.x
* **数据存储:** MySQL 8.x (持久化主库)
* **缓存与流控:** Redis (分布式令牌桶、AC 自动机规则热加载)
* **消息总线:** RocketMQ (异步削峰、流转驱动、分布式事务)
* **多路召回引擎:** Elasticsearch (倒排检索)、Milvus (向量检索)
* **特征提取:** BGE-M3 (或其他同等级 Embedding 模型)

---

## 2. 评测任务管理与执行流控

本模块负责评测用例的分发，以及与**被测大模型**的交互。核心难点在于**全异步非阻塞流转**与**严格的全局 QPS 削峰**。

### 2.1 任务数据模型

* `eval_tasks` (评测任务表): 记录全局任务，包含 `task_id`, `status` (INIT, RUNNING, FINISHED), `total_cases`, `success_cases`。
* `eval_cases` (评测用例表): 记录具体每一条测试，包含 `case_id`, `task_id`, `prompt`, `response`, `eval_result` (评测中/安全/违规), `hit_rules`。

### 2.2 流式分发与执行管道 (Execution Pipeline)

严禁“批处理等待”，必须采用单条流式处理。

1. **任务打散:** 调度节点将新任务分解为 $N$ 条用例，分别发送到 RocketMQ 的 `Execution_MQ` Topic。
2. **全局限流 (Token Bucket):** * Worker 节点在发起大模型 HTTP 调用前，必须通过 **Redis Lua 脚本** 申请令牌（按照厂商 QPS 限制配置，如 50/s）。
* 必须确保获取令牌的操作是原子的。


3. **退避与重试 (Backpressure):**
* 若 Redis 令牌不足，或调用时遭遇大模型厂商 `429 Too Many Requests`，拦截异常。
* 利用 RocketMQ 的**延迟消息**（Delay Level，如 5秒/10秒后）将该用例发回 `Execution_MQ` 进行指数退避重试。


4. **结果入库:** 获取到被测模型的 `Response` 后，更新 MySQL 中对应 `case_id` 的状态，并将 `[case_id, response]` 投入 `L1_MQ` 开始安全判定。

---

## 3. 三层漏斗判定架构与流转

在获取到模型回复后，进入安全判定的核心漏斗。各个层级之间严格通过 RocketMQ 解耦，各司其职。

### 3.1 第一层：L1 极速字面量拦截 (AC 自动机)

* **消费 Topic:** `L1_MQ`
* **处理逻辑:** Worker 节点在内存中维护 Aho-Corasick 自动机（通过 Redis Pub/Sub 订阅规则秒级热更新）。
* **流转决策:**
* **命中红线词:** 触发短路，直接将结果（违规）回写 MySQL，生命周期结束。
* **未命中:** 放行，将数据投递至 `L2_MQ`。



### 3.2 第二层：L2 异构双路召回 (ES + Milvus)

* **消费 Topic:** `L2_MQ`
* **处理逻辑:** * 提取文本，并发请求 ES（BM25）与 Milvus（Cosine Similarity）。
* 在 Worker 内存中执行 **RRF (倒数秩融合)** 算法合并分数，消除量纲冲突。


* **流转决策:**
* **绝对安全 (双低分):** 触发短路，直接回写 MySQL（安全），生命周期结束。
* **实体硬命中 (ES 高分突刺):** 触发短路，直接回写 MySQL（违规），生命周期结束。
* **高分模糊区 (Milvus $\ge$ 0.82 等):** 提取命中特征的 `scenario_id`，去 MySQL 拉取完整的“父分片（判定准则）”。将原文本与判定准则组装，投递至 `L3_MQ`。



### 3.3 第三层：L3 认知裁决 (Judge LLM)

* **消费 Topic:** `L3_MQ`
* **处理逻辑:** 将上下文组装为 Prompt，调用 Judge LLM（此链路单独配置并发限流器，保护内部算力）。
* **终判:** 根据裁判模型输出的 Reasoning 和最终定性，回写 MySQL 记录并完结该用例。
* **归档:** 依赖定时任务或 MQ 事务机制，当 `eval_tasks` 下所有 case 结束时，生成统计报表。

## 4. 知识库设计与数据同步机制

知识库是 L2 判定的弹药库。为保障架构极简与数据一致性，采用 **父子关联模型** 与 **RocketMQ 事务消息** 实现强一致性同步（取代 CDC 组件）。

### 4.1 数据模型设计 (MySQL)

* **`risk_scenarios` (父分片):** 存储宏观场景与 Judge LLM 使用的判决准则。
* **`attack_features` (子分片):** 存储用于检索的具体特征词、Payload 样本。

### 4.2 数据同步方案 (RocketMQ 事务消息)

对于知识库的任何新增/修改，必须统一走 MQ 事务 API，严禁直接调用 Mapper 写入。

1. **发送 Half 消息:** 管理后台发起更新，向 MQ 发送带有特征内容的半消息。
2. **执行本地事务:** 实现 `RocketMQLocalTransactionListener#executeLocalTransaction`。在该方法内开启 `@Transactional` 执行 MySQL 入库。
* 成功 $\rightarrow$ 返回 `COMMIT`。
* 失败 $\rightarrow$ 返回 `ROLLBACK`。


3. **事务回查兜底:** 实现 `checkLocalTransaction`。针对网络异常中断，通过 `SELECT` 查询本地 MySQL 确认数据是否真实落库，以补发 `COMMIT` 或 `ROLLBACK`。
4. **下游消费双写 (Worker 端):**
* 同步 Worker 消费确认后的消息。
* **幂等处理:** 根据 `feature_id` 进行操作。
* **顺序执行:** 调用 Embedding 模型取向量 $\rightarrow$ 使用 `feature_id` 进行 `UPSERT` 写入 Milvus $\rightarrow$ 使用 `feature_id` 执行 `PUT` 写入 ES。
* 若 Embedding 超时或下游组件异常，抛出异常依赖 MQ 重试（利用基于 ID 的 UPSERT 保证幂等覆盖）。
